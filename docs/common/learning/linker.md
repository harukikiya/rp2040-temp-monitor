# リンカスクリプト学習ノート

リンカスクリプトを書くために必要な基礎知識を、本プロジェクトの文脈に合わせてまとめる。
本プロジェクト固有の設計判断は ADR、リンカスクリプトファイル自体は
`firmware-c/src/linker/` 以下を参照。

## 1. リンカの役割

### ビルドの全体像

「ビルド」とは、ソースコードから最終成果物(実行可能バイナリ、ライブラリ等)を生成する
作業全体を指す。コンパイル、アセンブル、リンク、後処理など、複数の工程を含む。

C 言語のソースコードが実行可能なバイナリになるまでには、以下のステップを経る。

```
ソースコード (.c, .h, .S)
     ↓ プリプロセッサ (前処理)
前処理済みソース
     ↓ コンパイラ
アセンブリコード (.s)
     ↓ アセンブラ
オブジェクトファイル (.o)
     ↓ リンカ
実行可能 ELF (.elf)
     ↓ objcopy (後処理)
ファームウェアバイナリ (.bin, .uf2)
```

これら全体を「ビルド」と呼び、各ステップは「ビルドの工程」となる。

「コンパイル」という言葉は文脈によって意味が変わる。

- 狭義:C コード → アセンブリ(`clang -S`)
- 中義:C コード → オブジェクトファイル(`clang -c`)
- 広義:C コード → 実行可能バイナリ(`clang` 単体実行、内部でリンカも呼ばれる)

組込み開発では中義が一般的で、その後リンカが複数のオブジェクトファイルを結合して
最終バイナリにする。一連の流れ全体は「ビルド」と呼ぶのが正確。

### リンカが行う3つの仕事

リンカの仕事は大きく3つに分けられる。

#### 1. シンボル解決(Symbol Resolution)

複数のオブジェクトファイルにまたがる「この変数/関数はどこにあるか」を解決する。

シンボルの宣言と定義は、ヘッダファイル経由で行うのが推奨される書き方。

```c
// adc.h(ヘッダファイル、シンボルの宣言を一元管理)
#ifndef ADC_H
#define ADC_H
extern int adc_value;   // 宣言
int adc_read(void);     // プロトタイプ宣言(関数は extern 省略可)
#endif

// adc.c(実装)
#include "adc.h"
int adc_value;          // 定義(ここに実体がある)
int adc_read(void) {
    return adc_value;
}

// main.c(利用側)
#include "adc.h"        // ヘッダをインクルードするだけ
int main(void) {
    int x = adc_read();
    return adc_value;
}
```

このとき、`adc.c` をコンパイルすると `adc.o`、`main.c` をコンパイルすると `main.o`
ができる。`main.o` には「`adc_value` と `adc_read` を参照しているが、これらの実体は
別のオブジェクトファイルにある」という未解決情報が含まれる。

リンカが両方の `.o` を読み込み、「`adc_value` は `adc.o` の中の特定の位置にある」
と紐付けるのがシンボル解決。

ヘッダファイル経由で宣言を一元管理する利点:

- 型を変更する時、ヘッダ1箇所の修正で全利用側に反映される
- ヘッダがモジュールの API を明示する役割を持つ
- コンパイラが宣言と定義の型を照合し、不一致を検出できる

#### 2. 再配置(Relocation)

オブジェクトファイルの中では、関数や変数のアドレスは仮の値(または相対表現)で
書かれている。

リンカが「`adc_value` は最終的にアドレス `0x20000100` に配置される」と決めたら、
そのアドレスを参照する命令の `pc` 相対オフセットなどを、実際の値に書き換える。

これが再配置。「仮アドレスを実アドレスに書き換える」作業。

#### 3. セクション配置(Section Placement)

各オブジェクトファイルには、複数のセクションという単位でコードやデータが含まれる。

代表的なセクション:

- `.text`:プログラムコード(実行命令)。「text」は伝統的にコードを意味する
- `.rodata`:Read-Only Data の略。読み取り専用データ(文字列リテラル、`const` 変数)
- `.data`:初期値ありの可変データ(`int x = 5;` のような変数)
- `.bss`:Block Started by Symbol の略(1950年代の IBM 704 アセンブラに由来する歴史的名称)。
  初期値なし(または初期値が0)の可変データ(`int x;` のような変数)。
  起動時に 0 で初期化される

`.bss` セクションは ELF ファイルの中ではサイズだけ記録され、値そのものは記録されない
(すべて 0 だから)。起動時に crt0 が「このサイズ分の領域を 0 で埋める」処理を行う。
このため、`.bss` を使うとファームウェアのバイナリサイズが小さくなる。

複数の `.o` ファイルから、同じ名前のセクション(全ファイルの `.text` など)を
1つにまとめて、メモリ上のどこに配置するかを決めるのもリンカの仕事。

「どこに配置するか」を指示するのがリンカスクリプト。

### リンカが原因で発生する典型的なバグ

リンカは「複数のオブジェクトファイルを結合する」という性質上、独特なバグを生む可能性がある。
代表的なものを挙げる。

#### 1. 同名のグローバル変数を複数ファイルで定義

```c
// file_a.c
int counter = 0;

// file_b.c
int counter = 5;  // 別ファイルで同名定義
```

シンボル `counter` が複数定義され、多くのリンカは「multiple definition of 'counter'」
エラーを出す。

対策:

- 共通の変数なら、1箇所だけで定義し、他は extern 宣言(ヘッダ経由)
- ファイル内に閉じる変数は `static` を付けて外部から見えなくする

#### 2. 宣言と定義の型不一致

```c
// adc.h
extern int adc_value;   // int として宣言

// adc.c
short adc_value;        // short として定義(型が違う)
```

ヘッダ経由で `int` だと思っている利用側が、実際には `short` のメモリを `int` として
読み書きしてしまう。結果として未定義動作(データ破損、隣接データの書き換え)が起きる。

怖い点として、コンパイラもリンカもこれを検出できない場合がある(両方とも自分の `.c`
ファイルの中だけ見ているため)。

対策:ヘッダ経由で宣言を一元化(`adc.c` も同じヘッダを include することで、型不一致が
コンパイル時にエラーとなる)。

#### 3. ガベージコレクションによる必要セクションの削除

組込みではコードサイズ削減のため、リンク時に未使用関数/データを削除する `--gc-sections`
オプションがよく使われる。ただし、CPU がアドレスで直接アクセスするセクション
(ベクタテーブルなど)は「どの関数からも参照されない」ように見えるため、誤って削除される
可能性がある。

対策:リンカスクリプトで `KEEP()` ディレクティブを使って「削除してはいけない」と
明示する。詳細は後の「ガベージコレクション」セクションで扱う。

### ホスト開発と組込み開発の違い

#### ホスト開発(普通の Linux/Mac/Windows のアプリ)

- リンカは標準ライブラリ(`libc` など)を自動的に組み込む
- リンカスクリプトはほぼ意識しない(デフォルトのものが使われる)
- 実行時に OS のローダがバイナリをメモリにロードして実行
- ローダが ASLR(Address Space Layout Randomization)などを担当

#### 組込み開発(本プロジェクト、ベアメタル)

- 本プロジェクトはベアメタル(OS なし)で標準ライブラリも使わない
- リンカに `-nostdlib` を指示して、標準ライブラリの自動組み込みを抑制
- 必要な関数は自分で実装するか、newlib-nano などの組込み向け軽量ライブラリを使う
- リンカスクリプトを自分で書く必要がある(ハードウェアごとにメモリレイアウトが違うため)
- 実行時に OS はない。boot2 + crt0 がバイナリを正しく動かす役割を担う
- メモリアドレスは固定(0x10000000 から始まる、など)

なお、組込み開発でも RTOS(FreeRTOS、Zephyr など)を使う場合は、RTOS の流儀に従って
標準ライブラリを組み込むことがある。本プロジェクトでは RTOS は扱わない(ベアメタル一本)。

つまり、組込み開発ではリンカが「OS のローダの代わり」になる準備をする。
これが組込みでリンカスクリプトが重要な理由。

### 本プロジェクトでのリンカの位置づけ

本プロジェクトのビルド流れ(将来):

```
.c ファイル (Clang でコンパイル) → .o
.S ファイル (Clang でアセンブル) → .o (boot2 等)
                ↓
       リンカ + リンカスクリプト (rp2040.ld)
                ↓
       firmware.elf
                ↓
       objcopy → firmware.bin
                ↓
       elf2uf2 → firmware.uf2 (RP2040 ブートローダがこれを受け取る)
```

リンカが間違うと、boot2 が正しい位置に配置されない、ベクタテーブルが見つからない、
データが破損する、などの問題が起きる。
リンカスクリプトの設計は、組込みファームウェアの動作の前提となる。
