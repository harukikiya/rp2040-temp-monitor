# リンカスクリプト学習ノート

リンカスクリプトを書くために必要な基礎知識を、本プロジェクトの文脈に合わせてまとめる。
本プロジェクト固有の設計判断は ADR、リンカスクリプトファイル自体は
`firmware-c/src/linker/` 以下を参照。

## 1. リンカの役割

### ビルドの全体像

「ビルド」とは、ソースコードから最終成果物(実行可能バイナリ、ライブラリ等)を生成する
作業全体を指す。コンパイル、アセンブル、リンク、後処理など、複数の工程を含む。

C 言語のソースコードが実行可能なバイナリになるまでには、以下のステップを経る。

```
ソースコード (.c, .h, .S)
     ↓ プリプロセッサ (前処理)
前処理済みソース
     ↓ コンパイラ
アセンブリコード (.s)
     ↓ アセンブラ
オブジェクトファイル (.o)
     ↓ リンカ
実行可能 ELF (.elf)
     ↓ objcopy (後処理)
ファームウェアバイナリ (.bin, .uf2)
```

これら全体を「ビルド」と呼び、各ステップは「ビルドの工程」となる。

「コンパイル」という言葉は文脈によって意味が変わる。

- 狭義:C コード → アセンブリ(`clang -S`)
- 中義:C コード → オブジェクトファイル(`clang -c`)
- 広義:C コード → 実行可能バイナリ(`clang` 単体実行、内部でリンカも呼ばれる)

組込み開発では中義が一般的で、その後リンカが複数のオブジェクトファイルを結合して
最終バイナリにする。一連の流れ全体は「ビルド」と呼ぶのが正確。

### リンカが行う3つの仕事

リンカの仕事は大きく3つに分けられる。

#### 1. シンボル解決(Symbol Resolution)

複数のオブジェクトファイルにまたがる「この変数/関数はどこにあるか」を解決する。

シンボルの宣言と定義は、ヘッダファイル経由で行うのが推奨される書き方。

```c
// adc.h(ヘッダファイル、シンボルの宣言を一元管理)
#ifndef ADC_H
#define ADC_H
extern int adc_value;   // 宣言
int adc_read(void);     // プロトタイプ宣言(関数は extern 省略可)
#endif

// adc.c(実装)
#include "adc.h"
int adc_value;          // 定義(ここに実体がある)
int adc_read(void) {
    return adc_value;
}

// main.c(利用側)
#include "adc.h"        // ヘッダをインクルードするだけ
int main(void) {
    int x = adc_read();
    return adc_value;
}
```

このとき、`adc.c` をコンパイルすると `adc.o`、`main.c` をコンパイルすると `main.o`
ができる。`main.o` には「`adc_value` と `adc_read` を参照しているが、これらの実体は
別のオブジェクトファイルにある」という未解決情報が含まれる。

リンカが両方の `.o` を読み込み、「`adc_value` は `adc.o` の中の特定の位置にある」
と紐付けるのがシンボル解決。

ヘッダファイル経由で宣言を一元管理する利点:

- 型を変更する時、ヘッダ1箇所の修正で全利用側に反映される
- ヘッダがモジュールの API を明示する役割を持つ
- コンパイラが宣言と定義の型を照合し、不一致を検出できる

#### 2. 再配置(Relocation)

オブジェクトファイルの中では、関数や変数のアドレスは仮の値(または相対表現)で
書かれている。

リンカが「`adc_value` は最終的にアドレス `0x20000100` に配置される」と決めたら、
そのアドレスを参照する命令の `pc` 相対オフセットなどを、実際の値に書き換える。

これが再配置。「仮アドレスを実アドレスに書き換える」作業。

#### 3. セクション配置(Section Placement)

各オブジェクトファイルには、複数のセクションという単位でコードやデータが含まれる。

代表的なセクション:

- `.text`:プログラムコード(実行命令)。「text」は伝統的にコードを意味する
- `.rodata`:Read-Only Data の略。読み取り専用データ(文字列リテラル、`const` 変数)
- `.data`:初期値ありの可変データ(`int x = 5;` のような変数)
- `.bss`:Block Started by Symbol の略(1950年代の IBM 704 アセンブラに由来する歴史的名称)。
  初期値なし(または初期値が0)の可変データ(`int x;` のような変数)。
  起動時に 0 で初期化される

`.bss` セクションは ELF ファイルの中ではサイズだけ記録され、値そのものは記録されない
(すべて 0 だから)。起動時に crt0 が「このサイズ分の領域を 0 で埋める」処理を行う。
このため、`.bss` を使うとファームウェアのバイナリサイズが小さくなる。

複数の `.o` ファイルから、同じ名前のセクション(全ファイルの `.text` など)を
1つにまとめて、メモリ上のどこに配置するかを決めるのもリンカの仕事。

「どこに配置するか」を指示するのがリンカスクリプト。

### リンカが原因で発生する典型的なバグ

リンカは「複数のオブジェクトファイルを結合する」という性質上、独特なバグを生む可能性がある。
代表的なものを挙げる。

#### 1. 同名のグローバル変数を複数ファイルで定義

``` c
// file_a.c
int counter = 0;

// file_b.c
int counter = 5;  // 別ファイルで同名定義
```

シンボル `counter` が複数定義され、多くのリンカは「multiple definition of 'counter'」
エラーを出す。

対策:

- 共通の変数なら、1箇所だけで定義し、他は extern 宣言(ヘッダ経由)
- ファイル内に閉じる変数は `static` を付けて外部から見えなくする

#### 2. 宣言と定義の型不一致

```c
// adc.h
extern int adc_value;   // int として宣言

// adc.c
short adc_value;        // short として定義(型が違う)
```

ヘッダ経由で `int` だと思っている利用側が、実際には `short` のメモリを `int` として
読み書きしてしまう。結果として未定義動作(データ破損、隣接データの書き換え)が起きる。

怖い点として、コンパイラもリンカもこれを検出できない場合がある(両方とも自分の `.c`
ファイルの中だけ見ているため)。

対策:ヘッダ経由で宣言を一元化(`adc.c` も同じヘッダを include することで、型不一致が
コンパイル時にエラーとなる)。

#### 3. ガベージコレクションによる必要セクションの削除

組込みではコードサイズ削減のため、リンク時に未使用関数/データを削除する `--gc-sections`
オプションがよく使われる。ただし、CPU がアドレスで直接アクセスするセクション
(ベクタテーブルなど)は「どの関数からも参照されない」ように見えるため、誤って削除される
可能性がある。

対策:リンカスクリプトで `KEEP()` ディレクティブを使って「削除してはいけない」と
明示する。詳細は後の「ガベージコレクション」セクションで扱う。

### ホスト開発と組込み開発の違い

#### ホスト開発(普通の Linux/Mac/Windows のアプリ)

- リンカは標準ライブラリ(`libc` など)を自動的に組み込む
- リンカスクリプトはほぼ意識しない(デフォルトのものが使われる)
- 実行時に OS のローダがバイナリをメモリにロードして実行
- ローダが ASLR(Address Space Layout Randomization)などを担当

#### 組込み開発(本プロジェクト、ベアメタル)

- 本プロジェクトはベアメタル(OS なし)で標準ライブラリも使わない
- リンカに `-nostdlib` を指示して、標準ライブラリの自動組み込みを抑制
- 必要な関数は自分で実装するか、newlib-nano などの組込み向け軽量ライブラリを使う
- リンカスクリプトを自分で書く必要がある(ハードウェアごとにメモリレイアウトが違うため)
- 実行時に OS はない。boot2 + crt0 がバイナリを正しく動かす役割を担う
- メモリアドレスは固定(0x10000000 から始まる、など)

なお、組込み開発でも RTOS(FreeRTOS、Zephyr など)を使う場合は、RTOS の流儀に従って
標準ライブラリを組み込むことがある。本プロジェクトでは RTOS は扱わない(ベアメタル一本)。

つまり、組込み開発ではリンカが「OS のローダの代わり」になる準備をする。
これが組込みでリンカスクリプトが重要な理由。

### 本プロジェクトでのリンカの位置づけ

本プロジェクトのビルド流れ(将来):

```
.c ファイル (Clang でコンパイル) → .o
.S ファイル (Clang でアセンブル) → .o (boot2 等)
                ↓
       リンカ + リンカスクリプト (rp2040.ld)
                ↓
       firmware.elf
                ↓
       objcopy → firmware.bin
                ↓
       elf2uf2 → firmware.uf2 (RP2040 ブートローダがこれを受け取る)
```

リンカが間違うと、boot2 が正しい位置に配置されない、ベクタテーブルが見つからない、
データが破損する、などの問題が起きる。
リンカスクリプトの設計は、組込みファームウェアの動作の前提となる。

## 2. ELF ファイルの基本構造

### ELF とは何か

ELF は **Executable and Linkable Format** の略。1990年台に Unix System V のために
設計された、実行ファイル・オブジェクトファイル・共有ライブラリの標準フォーマット。
現代では Linux、BSD、組み込みシステムなど、ほぼすべての Unix 系 / 組み込み環境で使われている。

ELF が表現できるもの：

- **実行可能ファイル（Executable）**：実行できるバイナリ（例：`/bin/ls`）
- **オブジェクトファイル（Relocatable）**：コンパイル直後の `.o` ファイル
- **共有ライブラリ（Shared object）**：動的リンクライブラリ（例：`libc.so`）
- **コアダンプ（Core）**：クラッシュ時のメモリイメージ

本プロジェクトで関係するのは：

- **オブジェクトファイル**：`firmware-c/build/*.o`（コンパイル後）
- **実行可能ファイル**：`firmware-c/build/firmware.elf`（リンク後）

### なぜ ELF を理解する必要があるのか

リンカスクリプトを書くということは、「リンカに対して、ELF をどう構築するかを指示する」こと。

- `MEMORY` で「フラッシュは 0x10000000 から 2MB」と書く
  → ELF の各セクションがそのアドレス範囲に配置される
- `SECTIONS` で「`.text` はフラッシュに `.bss` は SRAM に」と書く
  → ELF にセクションとして含まれる、または含まれないものが決まる
- `KEEP()` で「このセクションは削除しないで」と書く
  → ELF のサイズや内容に影響する

ELF の中身を知らずにリンカスクリプトを書くと、「なぜこのオプションを指定するのか」
が分からないまま暗記することになる。

### ELF ファイルの全体構造

ELF ファイルは以下の構造を持つ。
 
```
+--------------------------------+
| ELF ヘッダ                    | ← ファイル先頭、固定サイズ
|                                |   (32-bit ELF で 52 バイト、64-bit ELF で 64 バイト)
+--------------------------------+
| プログラムヘッダテーブル        | ← 実行時にメモリにロードする情報
| (Program Header Table)         |   (実行ファイル・共有ライブラリで必要、.o では省略可)
+--------------------------------+
| セクション 1 (.text)           | ← 各セクションの実体データ
| セクション 2 (.rodata)         |
| セクション 3 (.data)           |
| セクション 4 (.bss) ※          |   ※ .bss はサイズ情報のみ、データは含まれない
| ...                            |
+--------------------------------+
| セクションヘッダテーブル        | ← 各セクションの位置・サイズ・属性情報
| (Section Header Table)         |   (リンク時に必要)
+--------------------------------+
```

### 3つの主要な「ヘッダ」

ELF を理解する鍵は、3種類のヘッダの違い。

#### 1. ELF ヘッダ（ELF Header）

ファイルの先頭にある、ファイル全体のメタ情報。以下を含む。

- マジックナンバー（`0x7F 0x45 0x4C 0x46` = `\x7F` + `ELF`）
- ELF クラス（32-bit / 64-bit、本プロジェクトは 32-bit）
- エンディアン（リトルエンディアン / ビッグエンディアン、本プロジェクトはリトルエンディアン）
- ファイルタイプ（実行可能 / オブジェクト / 共有ライブラリ）
- マシンアーキテクチャ（ARM、x86_64、RISC-V など、本プロジェクトは ARM）
- エントリポイントのアドレス（プログラム開始位置）
- プログラムヘッダテーブルの位置とサイズ
- セクションヘッダテーブルの位置とサイズ

#### 2. プログラムヘッダテーブル（Program Header Table）

実行時に重要。「ファイルのどの部分を、メモリのどこに、どう（読み取り専用 / 書き込み可など）ロードするか」を記述する。

各エントリを **セグメント（Segment）** と呼ぶ。

例：

- 「ファイルの 0x1000 〜 0x5000 を、メモリの 0x10000000 〜 0x100004000 にロード（読み取り + 実行可）」
- 「ファイルの 0x5000 〜 0x5100 を、メモリの 0x20000000 〜 0x200001000 にロード（読み書き可）」

#### 3. セクションヘッダテーブル（Section Header Table）

リンク時に重要。「ファイルのどの部分が、どのセクション（`.text`、`.data`、`.bss` など）に対応するか」を記述する。

各エントリを **セクション（Section）** と呼ぶ。

### セクションとセグメントの違い

ここが混乱しやすいポイント。同じデータが、リンク時はセクションとして、実行時はセグメント
として見えることがある。
 
```
リンク時の視点(セクション)        実行時の視点(セグメント)
+--------------------+              +--------------------+
| .text              |              |                    |
+--------------------+              | LOAD セグメント1   |
| .rodata            |   →          | (読み取り+実行可)   |
+--------------------+              |                    |
                                    +--------------------+
+--------------------+              +--------------------+
| .data              |              |                    |
+--------------------+              | LOAD セグメント2   |
| .bss               |   →          | (読み書き可)        |
+--------------------+              |                    |
                                    +--------------------+
```

- **セクション**：内容ごとの細かい分類（コード、データ、定数、デバッグ情報など）
- **セグメント**：実行時のロード単位（同じ属性のセクションをまとめたもの）

リンカは、リンカスクリプトに従って：

1. 入力された `.o` ファイルからセクションを集めて、出力 ELF のセクションを作る
2. 同じ属性（読み取り権限、実行権限）のセクションを束ねて、セグメントを作る
3. セクションヘッダテーブルとプログラムヘッダテーブルの両方を書き込む

組み込み向けの ELF では、プログラムヘッダテーブルが簡略化されているか、
ない場合もある（OS のローダが使うものなので、ベアメタルでは ctr0 が代わりを担う）。

### オブジェクトファイル `.o` と実行可能 ELF `.elf` の違い


両方とも ELF フォーマットだが、中身が違う。

| 項目 | オブジェクトファイル `.o` | 実行可能 ELF `.elf` |
| --- | ---| --- |
| ELF ファイルタイプ | `ET_REL`（再配置可能） | `ET_EXEC`（実行可能） |
| セクションのアドレス | 仮の値（`0x0` から始まる相対値） | 実アドレス（リンカスクリプトに従う） |
| シンボル | 未解決（`extern` で参照のみ） | 全て解決済み |
| 再配置情報 | 含まれる（`.rela.text` など） | 削除済み |
| プログラムヘッダテーブル | なし（リンク時には不要） | あり（実行時に必要） |

リンカは「再配置可能な `.o` を集めて、実行可能な `.elf` を生成する」装置。

### 補足：ペリフェラルアドレスとリテラルプール

組み込み開発でよく使うペリフェラルへのアクセスは、ELF の中でどう表現されるか。
そしてリンカは関与するのか。

#### ペリフェラルアドレスはハードウェアで固定

RP2040 のペリフェラル（GPIO、I2C、ADC、SIO など）は、それぞれ固定のアドレスにメモリマップされている。
例：

- I2C0: `0x40044000`
- ADC: `0x4004C000`
- SIO: `0xD0000000`

これらは RP2040 のシリコン設計時に決まった値で、ソフトウェアでは変更できない。
SVD ファイルに記述されており、本プロジェクトでは `RP2040.h` を介して参照する。

#### リテラルプールとは

ARM Cortex-M0+ は Thumb 命令セットを使い、命令長が 16 ビット固定。
そのため、命令の中に 32 ビットの即値を直接埋め込むことができない。

「命令の中に直接書けないなら、メモリの別の場所に値を置いておいて、そこからロードしよう」というアプローチがリテラルプール。

C コード：

``` c
volatile uint32_t *i2c_data = (uint_32_t *)0x40044040;
*i2c_data = 0;
```

コンパイラが生成するアセンブリ（概念）：

``` asm
some_function:
     ldr r1, [pc, #4]         ; PC + 4 の位置から 32 ビット値を読み込む
     movs r0, #0
     str r0, [r1]
     bx lr                    ; return
     .word 0x40044040         ; リテラルプール
```

`ldr r1, [pc, #4]` 命令は「PC からの相対オフセット 4 の位置から、32 ビット値を読んで `r1` に入れる」動き。
命令の少し後ろに置かれた `.word 0x40044040` を読む。

「プール」と呼ばれる理由は、複数の 32 ビット値（他のペリフェラルアドレス、関数のアドレス、大きな定数など）がまとめて配置されるため。

#### リテラルプールの場所

リテラルプールは関数の末尾近くに置かれる（コンパイラの判断）。長い関数の場合は、途中にも置かれる。
`ldr r?, [pc, #N]` の `N` の範囲制約（0〜1020）があるため、参照元から近い位置に配置する必要がある。

リテラルプールは `.text` セクションの一部。最終的にフラッシュに書き込まれる。

#### Cortex-M3 以降との違い

Cortex-M3、M4、M33 などの上位 Cortex-M シリーズは `MOVW` / `MOVT` 命令で
32 ビット即値を 2 命令で直接ロードできるため、リテラルプール経由でなくても良い。

Cortex-M0+ は `MOVW` / `MOVT` がないため、リテラルプール経由が必須。
将来 RP2350（Cortex-M33）への移植では、コンパイラがリテラルプールを使わなくなる可能性がある（コード生成が変わる）。

#### リンカは関与するか

リンカは関与しない。理由は以下のとおり。

1. ペリフェラルアドレスはハードウェアで固定されているため、ソフトウェア側で配置を決める対象ではない。
2. C コードでアドレスを参照すると、コンパイラがリテラルプールに値を置き、命令にロード処理を生成する。
3. リテラルプールは `.text` セクションの一部として、コンパイラ・アセンブラの段階で確定している。
4. リンカは `.text` セクション全体をフラッシュのどこに配置するか決めるが、その中身には触らない。

#### リンカスクリプトでの扱い

リンカスクリプトの `MEMORY` ディレクティブには、フラッシュと SRAM だけを置く。
ペリフェラル領域（`0x40000000` 以降など）は書かない。コードや変数をそこに配置しないため、リンカが意識する必要がない。

#### リテラルプールに置かれるのはペリフェラルアドレスだけではない

リテラルプールには、命令に直接埋め込めない以下のような値が置かれる。

- ペリフェラルアドレス（例：`0x40044040`）
- 大きな定数（例：`0x12345678`）
- 関数のアドレス（関数ポインタとして使う場合）
- グローバル変数のアドレス
- 文字列リテラルのアドレス（`.rodata` に置かれた文字列の場所）

#### objdump での観察例

将来 `firmware.elf` を逆アセンブルすると、リテラルプールがこのように見える。

``` text
00010048 <send_byte>:
     10048:    4901      ldr r1, [pc, #4]         ; 10050 <send_byte + 0x8>
     1004a:    7008      strb r0, [r1, #0]
     1004c:    4770      bx lr
     1004e:    46c0      nop                      ; アライメント調整
     10050:    40044040  .word 0x40044040         ; リテラルプール
```

`10050` の位置に `0x40044040` という値が格納されており、 `ldr r1, [pc, #4]` でそこから読み込んでいる。
`46c0     nop` はリテラルプールを 4 バイト境界に揃えるためのダミー命令。

### ELF を読むツール

ELF の中身を見るためのツールが、Arm GNU Toolchain に含まれている。

#### `readelf`

ELF の構造を人間が読める形で表示するツール。

``` bash
# ELF ヘッダを表示
arm-none-eabi-readelf -h firmware.elf

# セクション一覧を表示
arm-none-eabi-readelf -S firmware.elf

# プログラムヘッダ（セグメント）一覧を表示
arm-none-eabi-readelf -l firmware.elf

# シンボルテーブルを表示
arm-none-eabi-readelf -s firmware.elf
```

#### `objdump`

ELF を逆アセンブルしたり、各セクションの中身を表示したりするツール。

``` bash
# 全セクションのヘッダ情報
arm-none-eabi-objdump -h firmware.elf

# 逆アセンブル（.text セクションを ARM 命令に変換して表示）
arm-none-eabi-objdump -d firmware.elf

# 逆アセンブル + ソース混在表示（-g でデバッグ情報付きビルドが必要）
arm-none-eabi-objdump -S firmware.elf

# 全シンボル
arm-none-eabi-objdump -t firmware.elf
```

`objdump` の逆アセンブル（`-d`）は機械語 → アセンブリの変換で、ほぼ完璧に元通りになる。
ただし、機械語 → C ソースコードへの逆変換（逆コンパイル）はできない。
コメントや変数名の情報は機械語には含まれていないため。

`-S` オプションで「ソースコード混在表示」が可能だが、これはデバッグ情報（DWARF）に記録された
「このアドレスは元の C のここから生成された」という対応関係を利用するもので、逆コンパイルではない。
デバッグ情報なしの ELF では機能しない。

#### `nm`

シンボル一覧を表示するシンプルなツール。

``` bash
arm-none-eabi-nm firmware.elf
```

#### `size`

各セクションのサイズを表示。コードサイズを確認するときに便利。

``` bash
arm-none-eabi-size firmware.elf
# 出力例：
#    text      data      bss       dec       hex  filename
#    12345     100       500       12945     3291 firmware.elf
```

本プロジェクトでは、段階2-C-5 以降にこれらのツールを使って `firmware.elf` の中身を観察する予定。

## 3. リンカスクリプトの基本文法

### リンカスクリプトとは（復習）

リンカスクリプトは、リンカに対して「各セクションのメモリのどこに、どう配置するか」を指示するファイル。
拡張子は通常 `.ld`。リンカが `.o` を集めて `.elf` を作る際の、セクション配置の設計図となる。

### リンカスクリプトの2つの主要ブロック

リンカスクリプトの中心は2つのブロック。

``` text
MEMORY    { ... }   ← メモリ領域の定義（どんなメモリがどこにあるか）
SECTIONS  { ... }   ← セクションの配置（各セクションのどのメモリに置くか）
```

### MEMORY ブロック

ターゲットにどんなメモリがあり、それぞれどのアドレスから何バイトかを定義する。

#### 構文

``` text
MEMORY
{
     領域名（属性）: ORIGIN = 開始アドレス, LENGTH = サイズ
}
```

#### RP2040 での例

``` text
MEMORY
{
     FLASH (rx)     : ORIGIN = 0x10000000, LENGTH = 2048k
     RAM   (rwx)    : ORIGIN = 0x20000000, LENGTH = 264k
}
```

各要素の意味：

- `FLASH`、`RAM`：領域の名前（自分で付ける、慣例的に大文字）
- `(rx)`、`(rwx)`：属性
  - `r` = readable（読み取り可）
  - `w` = writable（書き込み可）
  - `x` = executable（実行可）
- `ORIGIN`：領域の開始アドレス（FLASH は XIP 領域 0x10000000、RAM は SRAM 0x20000000）
- `LENGTH`：領域のサイズ（`k` = 1024、`M` = 1024×1024。`2048k` = 2MB）

属性は主に警告のために使われる。書き込み可能なデータを `FLASH (rx)` に配置しようとするとリンカが警告を出すことがある。
実際のメモリ保護（書き込み禁止）は属性だけでは行われず、MPU（Memory Protection Unit）の別途設定が必要。

### SECTIONS ブロック

各セクションを、MEMORY で定義したどの領域に、どう配置するかを指示する。
リンカスクリプトの本体。

#### 構文の基本

``` text
SECTIONS
{
     出力セクション名 :
     {
          入力セクションの指定
     } > 配置先のメモリ領域
}
```

#### シンプルな例

``` text
SECTIONS
{
     .text :
     {
          *(.text)
     } > FLASH
}
```

- `.text` という出力セクションを作る
- `*(.text)` = 全ての入力ファイル（`*`）の `.text` セクションを集める
- それを `FLASH` 領域に配置（`> FLASH`）

#### `*(.text)` の意味

- `*` = 全ての入力オブジェクトファイル（ワイルドカード）
- `(.text)` = その中の `(.text)` セクション

特定ファイルだけ指定も可能：

``` text
boot2.o(.text)      # boot2.o の .text だけ
*(.text)            # 全ファイルの .text
```

`*(.text)` と `*(.text*)` の違い：

- `*(.text)`：ちょうど `.text` という名前のセクション
- `*(.text)`：`.text` で始まる名前のセクション（`.text.func1` など）

`-ffunction-sections` でコンパイルすると関数ごとに `.text.関数名` という別セクションになるため、
`.text*` で全部拾う必要がある。

### RP2040 のリンカスクリプトの骨格（簡略版）

``` text
MEMORY
{
     FLASH (rx)     : ORIGIN = 0x10000000, LENGTH = 2048k
     RAM   (rwx)    : ORIGIN = 0x20000000, LENGTH = 264k
}

SECTIONS
{
     /* boot2 をフラッシュの先頭に配置 */
     .boot2 :
     {
          KEEP(*(.boot2))
     } > FLASH

     /* ベクタテーブルを boot2 の直後に配置 */
     .vector_table :
     {
          KEEP(*(.vector_table))
     } > FLASH

     /* プログラムコード */
     .text :
     {
          *(.text)
          *(.text*)
     } > FLASh

     /* 読み取り専用データ */
     .rodata :
     {
          *(.rodata)
          *(.rodata*)
     } > FLASH

     /* 初期値ありデータ */
     .data :
     {
          *(.data)
          *(.data*)
     } > RAM AT> FLASH

     /* 初期値なしデータ */
     .bss :
     {
          *(.bss)
          *(.bss*)
     } > RAM
}
```

boot2 とベクタテーブルを別セクションにする理由：

- boot2 はフラッシュの先頭 256 バイト固定（RP2040 のブート ROM がそこを読む）
- ベクタテーブルは boot2 の直後に固定（CPU がそのアドレスを参照する）
- 配置順を厳密に制御するため、別セクションにして順番を明示する

### KEEP() ディレクティブ

``` text
.boot2 :
{
     KEEP(*(.boot2))
} > FLASH
```

`KEEP(...)` は「このセクションをガベージコレクション（`--gc-sections`）で削除しないで」という指示。

boot2 やベクタテーブルはコードから関数呼び出しで参照されない（CPUがハードウェア的に直接アクセスする）ため、
`--gc-sections` が「使われていない」と誤判断して削除する可能性がある。
それを防ぐのが `KEEP()` 。詳細は「7. ガベージコレクション」で扱う。

### `>` と `AT>` の違い（重要）

リンカスクリプトで最も重要かつ混乱しやすいポイント。

#### `>` （VMA、実行時アドレス）

``` text
.text : { *(.text) } > FLASH
```

`> FLASH` は「この出力セクションの VMA（実行時アドレス）を FLASH 領域にする」という意味。

VMA = Virtual Memory Address（実行時にこのセクションが存在するアドレス）。

#### `AT>`（LMA、ロード時アドレス）

``` text
.data : { *(.data) } > RAM AT> FLASH
```

`> RAM AT> FLASH` は2つの指定を含む：

- `> RAM`：VMA（実行時アドレス）は RAM
- `AT> FLASH`：LMA（ロード時アドレス）は FLASH

`.data` の特殊事情：

- `.data` は初期値ありの変数（例：`int x = 5;`）
- 初期値 `5` はフラッシュに保存される（電源を切っても消えないように） = LMA は FLASH
- 変数は書き換え可能なので実行時は RAM に存在する必要がある = VMA は RAM
- 起動時に crt0 がフラッシュ（LMA）から RAM（VMA）へ初期値をコピーする

`.text` は `.rodata` は LMA = VMA（両方フラッシュ）なので `AT>` は不要。

#### `>` と `AT>` の使い分けまとめ

| セクション | 配置 | 構文 |
| --- | --- | --- |
| `.text` | フラッシュで実行（LMA = VMA） | `> FLASH` |
| `.rodata` | フラッシュで読む（LMA = VMA） | `> FLASH` |
| `.data` | 初期値フラッシュ、実体 RAM（LMA ≠ VMA） | `> RAM AT> FLASH` |
| `.bss` | RAM のみ（初期値なし） | `> RAM` |

### ENTRY() ディレクティブ

``` text
ENTRY(_reset_handler)
```

`ENTRY(...)` は「プログラムの開始位置（エントリポイント）はこのシンボル」という指示。
ELF ヘッダの「エントリポイントアドレス」フィールドに書き込まれる。

組み込み（ベアメタル）では：

- ELF のエントリポイントはデバッガ（GDB）が使う
- 実際の起動は boot2 → ベクタテーブルのリセットハンドラという流れ（ELF エントリポイントとは別経路）
- `ENTRY()` はデバッグ時の便宜の意味合いが強い

### その他のよく使う要素

#### `.` （ロケーションカウンタ）

`.` は現在の配置位置を表す特殊なシンボル。

``` text
.  = ALIGN(4);      /* 現在位置を 4 バイト境界に揃える */
```

`ALIGN(4)` は 4 の倍数アドレスまで進める。アライメント調整に使う。

#### シンボル定義

リンカスクリプトの中でシンボル（アドレスを指す名前）を定義できる。

``` text
.data :
{
     _sdata = .;    /* .data の開始位置を _sdata に */
     *(.data)
     _edata = .;    /* .data の終了位置を _edata に */
} > RAM AT> FLASH
```

これらのシンボル（`_sdata`、`_edata` など）は crt0 の C コードから参照して
「どこからどこまでコピーするか」を知るために使う。詳細は「6. リンカが生成するシンボル」で扱う。

## 4. LMA vs VMA

### なぜこの概念が重要か

「3. リンカスクリプトの基本文法」で、 `.data : { *(.data) } > RAM AT> FLASH` という書き方を見た。
「初期値はフラッシュ、実体は RAM」という二重生活。この背後にあるのが LMA と VMA の分離という考え方。

組み込み開発で最も重要な概念の一つであり、同時に最も理解しづらい概念でもある。
ここを理解すると、 crt0 が何をするか（「5. .data 初期値転送」）が腑に落ちる。

### 2つのアドレスの定義

すべてのセクションは、2種類のアドレスを持つ。

#### VMA (Virtual Memory Address)

実行時に、そのセクションが存在するべきアドレス。

「Virtual」と付くが、仮想メモリ（OS のページング）とは無関係。
組み込みの文脈では「プログラムが実行されるときに、このデータ / コードがあると期待するアドレス」の意味。

例えば変数 `int x` が VMA `0x20000100` を持つなら、プログラムは「`x` は `0x20000100` にある」と思って動作する。
`x` を読み書きする命令は `0x20000100` にアクセスする。

#### LMA (Load Memory Address)

プログラムをロード（配置）するときに、そのセクションが置かれるアドレス。
「最初にどこに焼き込まれるか」「電源投入時にどこにあるか」を表す。

### なぜ2つに分かれるのか

多くのセクションでは LMA = VMA だが、一部のセクションでは LMA ≠ VMA になる。

#### LMA = VMA のケース（普通）

`.text`（プログラムコード）：

- LMA：フラッシュ（電源投入時、コードはフラッシュにある）
- VMA：フラッシュ（実行時も、コードはフラッシュにある。XIP で直接実行）
- LMA = VMA = フラッシュ

RP2040 は XIP（eXecute In Place）機能で、フラッシュ上のコードを直接実行できる。
なので `.text` はフラッシュに置きっぱなしで良い（コピー不要）。

`.rodata`（定数データ）：

- LMA：フラッシュ、VMA：フラッシュ（定数なので書き換え不要、フラッシュから直接読む）
- LMA = VMA = フラッシュ

#### LMA ≠ VMA のケース（特殊）

`.data`（初期値ありの変数）：`int x = 5;` のような変数を考える。

要件1：初期値 `5` はどこかに永続保存する必要がある。
電源を切っても初期値を覚えておく必要があり、RAM は電源を切ると消えるので、初期値はフラッシュに保存する。
→ LMA はフラッシュ

要件2：変数 `x` は実行時に書き換え可能でなければならない。
フラッシュは簡単に書き換えられない（特殊な手順が必要、遅い）ので、書き換え可能な変数は RAM に置く。
→ VMA は RAM

矛盾の解決：起動時にコピーする。
初期値 `5` はフラッシュ（LMA）に保存しておき、起動時に crt0 がフラッシュ（LMA）から RAM（VMA） へコピーする。
コピー後、プログラムは RAM 上の `x` を読み書きする。

つまり `.data` は「生まれはフラッシュ、住まいは RAM」という二重生活を送る。

#### `.bss` （初期値なしの変数）

`int y;`（初期値なし）のような変数：

- 初期値がない（正確には C 言語の仕様で 0 に初期化される）
- フラッシュに「0」を保存しておく必要はない（どうせ全部 0 だから）
- 起動時に crt0 が RAM 上の `.bss` 領域を 0 で埋めれば良い

なので：

- LMA：なし（フラッシュに実体を持たない）
- VMA：RAM
- 起動時に crt0 が 0 で初期化

「2. ELF ファイルの基本構造」で「`.bss` は ELF にサイズ情報のみ記録される」と説明したのは、
「初期値が全部 0 なので、わざわざフラッシュに 0 を並べて保存する必要がない、サイズだけ分かれば起動時に 0 で埋められる」という理由による。

### crt0 によるデータ初期化の流れ

起動時、crt0 が `.data` のコピーと `.bss` の 0 初期化を行う。

```{uml} ../../../diagrams/architecture/lma_vma_concept.puml
:caption: crt0 によるデータ初期化（.data コピーと .bss 初期化）
```

起動直後（crt0 実行前）は、`.data` の初期値はフラッシュにあるだけで、SRAM はまだ初期化されていない（不定値）。
crt0 が以下を実行することで、プログラムが期待する状態（`.data` に初期値、`.bss` が 0）が整う。

1. `.data` の初期値を、フラッシュ（LMA）から SRAM（VMA）へコピー
2. `.bss` 領域を 0 で初期化

### XIP がもたらす特殊性

一般的なマイコン（チップ内蔵フラッシュを持つもの）では、フラッシュにプログラム全体を焼き、
`.text` もフラッシュから直接実行する。

RP2040 の場合：

- 外付け QSPI フラッシュを使う（チップ内蔵フラッシュではない）
- boot2 が QSPI フラッシュを XIP モードに設定して、初めてフラッシュのコードが実行可能になる
- それ以前（boot2 自身）は、フラッシュの先頭 256 バイトが SRAM にコピーされて実行される

この「boot2 が XIP を有効化する」のが、boot2 の役割(段階2-C-2 で扱った内容)。
 
LMA/VMA の観点では:
 
- `.text`、`.rodata`:XIP 有効化後、フラッシュから直接実行/読み取り(LMA = VMA = フラッシュ)
- `.data`:初期値フラッシュ → crt0 が SRAM にコピー(LMA = フラッシュ、VMA = SRAM)
- `.bss`:crt0 が SRAM を 0 初期化(VMA = SRAM、LMA なし)
### よくある誤解
 
#### 誤解1:「VMA の Virtual は仮想メモリのこと」
 
組込みの LMA/VMA は、OS の仮想メモリ(ページング、MMU)とは無関係。単に
「ロード時アドレス」と「実行時アドレス」を区別する用語。「Virtual」は「実行時アドレス」
と読み替えると分かりやすい。
 
#### 誤解2:「`.text` も RAM にコピーして実行するのでは?」
 
RP2040 では XIP のおかげで `.text` はフラッシュから直接実行できるので、RAM への
コピーは不要。ただし、一部の特殊な関数(高速実行が必要、フラッシュアクセスを避けたい)
を意図的に RAM にコピーして実行することもある。その場合は専用セクション(例:
`.time_critical`)を作り、`> RAM AT> FLASH` で配置する(`.data` と同じ仕組み)。
本プロジェクトでは段階が進んでから検討する範囲。
 
#### 誤解3:「`.bss` もフラッシュに 0 を保存している」
 
`.bss` はフラッシュに実体を持たない(LMA なし)。全部 0 になることが分かっているので、
サイズ情報だけ持って、起動時に crt0 が RAM を 0 で埋める。これがバイナリサイズ節約になる。
 
### まとめ
 
1. すべてのセクションは LMA(ロード時アドレス)と VMA(実行時アドレス)を持つ
2. `.text`、`.rodata` は LMA = VMA(フラッシュ、XIP で直接実行/読み取り)
3. `.data` は LMA ≠ VMA(初期値フラッシュ、実体 RAM、crt0 がコピー)
4. `.bss` は VMA のみ(RAM、crt0 が 0 初期化、LMA なし)
5. crt0 が「`.data` のコピー」と「`.bss` の 0 初期化」を行うことで、プログラムが
   期待する状態を作る
6. 「Virtual」は仮想メモリとは無関係、「実行時アドレス」の意味

## 5・6. .data 初期値転送の仕組みとリンカが生成するシンボル

### 問題設定：crt0 が知る必要のあること

「4. LMA vs VMA」で、crt0 が起動時に2つの仕事をすると学んだ。

1. `.data` の初期値を、フラッシュ（LMA）から SRAM（VMA） へコピー
2. `.bss` 領域を 0 で初期化

crt0 を C 言語で書くとして、概念的には以下のようになる。

``` c
void crt0_init(void) {
     // （1）.data のコピー
     memcpy(/* コピー先（SRAM） */, /* コピー元（フラッシュ） */, /* サイズ */);
     // （2）.bss の 0 初期化
     memset(/* 対象（SRAM） */, 0, /* サイズ */);
}
```

ここで問題がある。コピー先（SRAM 上の `.data` 開始アドレス）、
コピー元（フラッシュ上の `.data` 初期値の開始アドレス）、
コピーサイズ、`.bss` の開始と終了--これらは、リンカがセクションを配置して初めて決まる値。
crt0 を書く時点までは未確定。

では、どうやって crt0 はこれらの値を知るのか。
答え：リンカがシンボルを生成して、crt0 に教える。

### リンカが生成するシンボルとは

リンカスクリプトの中でシンボル（アドレスを指す名前）を定義できる。

``` text
.data :
{
     _sdata = .;         /* .data の開始位置を _sdata に */
     *(.data)
     _edata = .;         /* .data の終了位置を _edata に */
} > RAM AT> FLASH
```

`_sdata` や `_edata` は、リンカが配置を決めた後のアドレス値を持つシンボル。
`.`(ロケーションカウンタ)は「現在の配置位置」を表すので、`_sdata = .;` は
「今まさに `.data` を配置しようとしている位置を `_sdata` という名前で覚えておく」の意味。
 
### 必要なシンボル一覧
 
慣例的な命名(`s` = start、`e` = end、`i` = init)で整理する。
 
#### `.data` 用
 
| シンボル | 意味 | アドレスの種類 |
|---|---|---|
| `_sdata` | `.data` の実体の開始(SRAM) | VMA |
| `_edata` | `.data` の実体の終了(SRAM) | VMA |
| `_sidata` | `.data` の初期値の開始(フラッシュ) | LMA |
 
- `_sdata` 〜 `_edata`:コピー先(SRAM 上)の範囲
- `_sidata`:コピー元(フラッシュ上)の開始位置
- コピーするサイズは `_edata - _sdata`
#### `.bss` 用
 
| シンボル | 意味 | アドレスの種類 |
|---|---|---|
| `_sbss` | `.bss` の開始(SRAM) | VMA |
| `_ebss` | `.bss` の終了(SRAM) | VMA |
 
- `_sbss` 〜 `_ebss`:0 で埋める範囲(SRAM 上)
- `.bss` は初期値を持たないので、コピー元(LMA)は不要。0 で埋めるだけ
### リンカスクリプトでのシンボル定義
 
```
SECTIONS
{
    /* ... boot2, .vector_table, .text, .rodata は省略 ... */
 
    .data :
    {
        . = ALIGN(4);
        _sdata = .;          /* .data 実体の開始(VMA, SRAM) */
        *(.data)
        *(.data*)
        . = ALIGN(4);
        _edata = .;          /* .data 実体の終了(VMA, SRAM) */
    } > RAM AT> FLASH
 
    _sidata = LOADADDR(.data);   /* .data 初期値の開始(LMA, フラッシュ) */
 
    .bss :
    {
        . = ALIGN(4);
        _sbss = .;           /* .bss の開始(VMA, SRAM) */
        *(.bss)
        *(.bss*)
        *(COMMON)
        . = ALIGN(4);
        _ebss = .;           /* .bss の終了(VMA, SRAM) */
    } > RAM
}
```
 
新しく出てきた要素:
 
- `LOADADDR(セクション名)`:そのセクションの LMA(ロード時アドレス)を返す。
  `.data` の VMA は `_sdata`(SRAM 上)だが、LMA はフラッシュ上の別の場所。
  `LOADADDR(.data)` でその LMA(初期値が置かれているフラッシュ上のアドレス)を
  取得して `_sidata` に入れる。これがコピー元になる
- `*(COMMON)`:初期化されていないグローバル変数の特殊な扱い(歴史的経緯)。
  `.bss` と同じ扱いなので一緒に拾う
- `. = ALIGN(4);`:各セクションの開始・終了を 4 バイト境界に揃える。
  ARM は 4 バイト単位のアクセスが効率的で、コピー処理を word 単位で行うと高速
### crt0 から見たシンボルの使い方
 
#### C コードでの宣言
 
シンボルは「リンカが定義したアドレス」なので、C コードでは `extern` で宣言する。
 
```c
extern uint32_t _sdata;    // .data 実体の開始(SRAM)
extern uint32_t _edata;    // .data 実体の終了(SRAM)
extern uint32_t _sidata;   // .data 初期値の開始(フラッシュ)
extern uint32_t _sbss;     // .bss の開始(SRAM)
extern uint32_t _ebss;     // .bss の終了(SRAM)
```
 
#### crt0 の初期化コード
 
```c
void crt0_init(void) {
    // (1) .data の初期値をフラッシュ(LMA)から SRAM(VMA)へコピー
    uint32_t *src = &_sidata;   // コピー元(フラッシュ)
    uint32_t *dst = &_sdata;    // コピー先(SRAM)
    while (dst < &_edata) {
        *dst++ = *src++;
    }
 
    // (2) .bss を 0 で初期化
    dst = &_sbss;
    while (dst < &_ebss) {
        *dst++ = 0;
    }
 
    // この後、main() を呼ぶ
}
```
 
#### なぜ `&`(アドレス演算子)を使うのか
 
`extern uint32_t _sdata;` と宣言したのに `&_sdata` とアドレスを取るのは、
リンカシンボルの特殊な扱いによる。
 
- リンカが定義する `_sdata` は「アドレスそのもの」(値ではない)
- C 言語の `extern uint32_t _sdata;` は「`_sdata` という名前の uint32_t 変数が
  どこかにある」という宣言
- リンカシンボルの場合、「変数のアドレス(`&_sdata`)」が「リンカが設定したアドレス値」
  に対応する
つまり:
 
- `_sdata`(値として読む)→ そのアドレスに格納されている内容(意味のない値)
- `&_sdata`(アドレスとして読む)→ リンカが設定したアドレス ← これが欲しい
「リンカシンボルはアドレスを `&` で取り出す」と覚える。
 
別の流儀として、配列として宣言すると `&` が不要になる。
 
```c
extern uint32_t _sdata[];   // 配列として宣言
uint32_t *dst = _sdata;     // 配列名はそのままアドレスになる(& 不要)
```
 
本プロジェクトでどちらを使うかは、crt0 を書く段階2-C-4 で決める。
 
### 全体の流れ
 
リンカがセクションを配置してアドレスを確定 → そのアドレスをシンボルとして埋め込む
→ crt0 がシンボルを参照してコピー処理を行う。

 
### なぜこの仕組みが必要なのか
 
- crt0 を書く時点では、`.data` や `.bss` の具体的なアドレス・サイズは未確定
- リンカがセクションを配置して初めて、アドレスが決まる
- リンカは決まったアドレスを「シンボル」として ELF に埋め込む
- crt0 はシンボルを `extern` で参照し、`&` でアドレスを取り出して、コピー/初期化を行う
この仕組みによって、crt0 はハードコードされたアドレスを書かずに、リンカスクリプトの
変更に追従できる。例えば SRAM のサイズが変わっても、リンカスクリプトを直せば crt0 は
そのまま動く(シンボル経由でアドレスを取るため)。
 
### 補足:なぜ memcpy / memset を使わず while ループで書くか
 
ベアメタルの起動初期では、標準ライブラリ(`memcpy` 等)がまだ使えない可能性がある。
特に `.data` のコピーが終わる前に `memcpy` を呼ぶと、`memcpy` 自身が `.data` を
使っていた場合に問題が起きる。そのため、依存のない素朴な while ループで書くのが安全。
(十分注意すれば `memcpy` を使う実装もある)
