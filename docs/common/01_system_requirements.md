# システム要件

## 機能要件

````{sysreq} 内蔵温度センサの継続的な可視化
:id: SYS_001
:status: approved
:type_kind: Functional
:tbd_items: TBD-001 表示周期500msは仮置き、実機での視認性確認後に調整。TBD-002 サンプリング周期100Hzも仮置き、温度センサの応答時間とリングバッファサイズのバランスで再評価。

本システムはRP2040の内蔵温度センサからの値を100Hzでサンプリングし、
LCD1602に温度として500ms周期で表示すること。
````

````{sysreq} 温度の統計表示
:id: SYS_002
:status: approved
:type_kind: Functional
:tbd_items: TBS-003 統計対象期間10秒は仮置き、実機動作確認後に調整。

本システムは直近10秒間の温度サンプル群から、最新値・最大値・最小値・平均値を計算しLCD1602に表示すること。
````

````{sysreq} 温度変化のトレンド表示
:id: SYS_003
:status: approved
:type_kind: Functional
:tbd_items: TBD-004 トレンド判定期間1分は仮置き。TBD-005 上昇下降の判定閾値は未定、実機での温度変動幅を観察して決める。

本システムは直近1分間の温度変化から上昇傾向・下降傾向・横ばいを判定し、LCD1602に記号で表示すること。
````

## 性能要件

````{sysreq} 応答性確保のためのコア分散
:id: SYS_004
:status: draft
:type_kind: Timing
:tbd_items: TD-006 統計計算の最大処理時間は実機計測後に確定。
:rationale: 現状の処理量（100Hzサンプリングは大半をDMAが担い、500ms周期の統計計算は8192サンプルの平均程度）では両コアとも余裕がある。それでもデュアルコア構成を採るのは、(1)RP2040のデュアルコア活用そのものを学習目的として明示している、(2)将来の機能追加(より複雑なフィルタ、通信機能等)に対する設計マージン、(3)サンプリング系と計算系を物理的に分離することで応答性を確保する、の3点による。

サンプリング処理とデータ処理を別コアで並列実行すること。
これにより、データ処理の負荷増加によるサンプリング欠落を防止する。

- Core0: サンプリング（ADC/DMA）、表示（I2C/LCD）
- Core1: 統計計算、トレンド判定
````

````{sysreq} コア間データ整合
:id: SYS_005
:status: draft
:type_kind: Safety

コア間で共有されるデータは、競合状態が発生しない方法で受け渡されること。
SIO FIFOまたはハードウェアスピンロックで保護されたバッファを使用する。
読み手が一貫したスナップショットを得られることを保証すること。
````

````{sysreq} I2C通信障害の検出と通知
:id: SYS_006
:status: draft
:type_kind: Safety
:tbd_items: TBD-007 連続失敗の閾値回数は仮置きで5回、実機で調整。TBD-008 LED点滅パターンは未定、エラー種別との対応含めて検討。

本システムはLCDとI2C通信が連続して失敗した場合、これを検出しオンボードLEDの点滅パターンによってユーザに通知すること。
通信失敗中もシステムは継続動作し、I2C通信が回復した時点で表示を再開すること。
````
