# ADR-0002: コア間計算結果の受け渡し機構の選択

## ステータス

採択

## 文脈

RP2040温度モニタプロジェクトのデュアルコア構成(SYS_004)において、
Core1で計算した統計結果(最新値、最大、最小、平均、トレンド判定結果)を
Core0へ受け渡す機構を選択する必要がある。

データの性質:
- 構造体サイズ: 約20バイト
- 更新頻度: 500ms周期
- 用途: LCD表示

要求される整合性レベル:
- 構造体原子性(レベル2)が必要。一部のフィールドだけ新しく、
  残りが古い中間状態が読まれると表示が論理矛盾する。
- 論理的整合性(レベル3)は計算側で自然に保証されるため、
  構造体原子性が確保されれば自動的に満たされる。

なお、Core0からCore1へのリングバッファ受け渡し(サンプル列)は
SWR_008の規定により最大1サンプル不整合を許容する設計であり、
本ADRの対象外である。

## 検討した選択肢

### 選択肢A: SIO FIFOで構造体を1ワードずつ送る

統計結果を32bitワード単位に分割し、SIO FIFOで順次送信。受信側が
複数ワードを組み立てて構造体として復元する。

利点:
- ハードウェアが原子性を保証(SIO FIFOはメッセージ単位で原子的)
- ソフトウェアによるロック実装の不具合リスクがない
- 実装がシンプル
- イベント駆動構造(SWR_024)と自然にフィットする

欠点:
- 構造体サイズに比例した送信時間
- FIFO容量(8段、各32bit)を超える大きさは送れない
- 送信側が満杯時にブロックする可能性

### 選択肢B: 共有メモリ + Spinlockで保護

統計結果を共有メモリ領域に置き、書き手と読み手の両方がSpinlockを
取って排他制御する。

利点:
- データサイズに制限なし
- 組込みでの一般的な手法

欠点:
- Lock取得待ちによるブロッキング
- Spinlock解放忘れによるデッドロックリスク
- 解放を確実にするRAII的な仕組みがCではないため、実装ミスの余地がある

### 選択肢C: ダブルバッファ + ポインタ切替

2つの構造体バッファを用意し、書き手は片方に書き込み、書き終わったら
有効バッファのポインタを切り替える。読み手は常に「有効ポインタ」が
指すバッファを読む。

利点:
- ロックなし、書き手と読み手が干渉しない
- ポインタ切替が4バイト原子操作で完了

欠点:
- メモリを2倍使う
- 読み手が古いバッファを読んでいる最中に書き手が次の書き込みを
  始めると競合の余地がある(完全な解決にはtriple bufferingが必要)
- 実装が複雑

### 選択肢D: SIO FIFOで更新通知、データは共有メモリ

書き手が共有メモリに書き終わったらSIO FIFOで通知ワードを送る。
読み手は通知を受けてから共有メモリを読む。

利点:
- 構造体サイズに制限なし
- 通知と実データを分離

欠点:
- 通知と実データの間に競合の余地がある(通知後・読み出し中に
  次の書き込みが始まると壊れる)
- 完全な解決には結局Spinlockかダブルバッファが必要
- 機構が複雑

## 決定

選択肢Aを採択する。SIO FIFOで構造体を1ワードずつ分解送信し、
受信側で組み立てて復元する。

## 理由

本システムの特性に最もフィットする:

1. **ハードウェア保証による安全性**: SIO FIFOはRP2040のハードウェアが
   メッセージ単位の原子性を保証する。ソフトウェアによる実装ミスの
   余地が少ない。

2. **データ量と頻度**: 20バイト構造体を5ワードに分けて送信しても
   500ms周期に対して十分高速。データ量制限の問題は発生しない。

3. **学習価値**: SIO FIFOというRP2040固有の機構を実体験できる。
   本プロジェクトの学習目的(SYS_004 rationaleで明記)に合致する。

4. **イベント駆動構造との親和性**: SIO FIFOは受信側にIRQを発生させる
   ことができ、SWR_024のイベント駆動構造に自然に組み込める。

5. **実装のシンプルさ**: Spinlock解放忘れやダブルバッファのタイミング
   競合といった実装上の落とし穴が少ない。

選択肢B、Cも組込みで一般的な手法であり、より大きなデータや高頻度な
更新では適切な選択となる。本プロジェクトではそのような状況に該当しない。

## 影響

ポジティブな影響:
- ロックフリーで実装ミスのリスクが少ない
- SIO FIFOの理解が深まる(学習価値)
- イベント駆動構造に自然に組み込める

ネガティブな影響(技術的負債):
- 将来構造体サイズが大幅に増えた場合(数百バイトを超えるなど)、
  選択肢Bや選択肢Dへの移行が必要になる可能性
- FIFO満杯時の挙動を別途決める必要がある(SWR_030のTBD-016)

将来の見直し条件:
- 計算結果の構造体サイズがFIFO 8段(32バイト)を超える時
- 更新頻度が大幅に上がる時(例: 10ms周期以下)
- Rust移植時にロックフリーデータ構造のクレートを採用する場合
