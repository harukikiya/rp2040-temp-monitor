# ADR-0003: リンカスクリプトの命名規約とデータ初期化方式の選択

## ステータス

採択

## 文脈

RP2040温度モニタプロジェクトのベアメタル基盤（段階2-C-3）において、
リンカスクリプト `firmware-c/src/linker/rp2040.ld` を自前実装するにあたり、
3つの設計判断が必要となった。

1. **リンカシンボルの命名規約**: `.data` のコピー元・コピー先や `.bss` の範囲、
   スタック頂点などを指すシンボルに、どの命名体系を使うか。学習ノート
   （`docs/common/learning/linker.md`）では教科書的な `_sdata` / `_edata` /
   `_sbss` / `_ebss` / `_stack_top` といった名前で解説しているが、実装でも
   同じ名前を使うかどうか。

2. **データ初期化方式**: 起動時に crt0（段階2-C-4で実装予定）が `.data` の
   初期値をフラッシュからRAMへコピーし `.bss` をゼロクリアする際、リンカ
   スクリプトが提供するシンボルの形式。単一の領域を前提とした単純シンボル
   方式か、複数領域に対応できるテーブル駆動方式か。

3. **メモリ領域の宣言サイズ**: `MEMORY` ブロックで FLASH / RAM をどう宣言
   するか。実サイズで宣言するか、将来の領域予約やSRAM分割を見越した
   宣言にするか。

これらは crt0（段階2-C-4）が参照するインターフェースを規定するため、
crt0実装の前に確定しておく必要がある。

## 検討した選択肢

### 判断1: リンカシンボルの命名規約

#### 選択肢1-A: 学習ノートと同じ教科書的命名（`_sdata` 系）

学習ノートで使った `_sdata` / `_edata` / `_sbss` / `_ebss` / `_stack_top` を
そのまま実装でも使う。

利点:
- 学習ノートと実装のシンボル名が一致し、対応関係を考えずに読める
- "start of data" / "end of data" という意味が名前から直感的

欠点:
- 実務・既存資産との整合性が低い。CMSIS や多くのベンダ提供スタートアップ
  コードは別の命名（`__data_start__` 系）を使っており、それらを参照・流用
  する際に名前の読み替えが必要になる
- 本プロジェクトの命名規約（ハンドラ・シンボルともCMSIS準拠）と不一致

#### 選択肢1-B: CMSIS準拠の命名（`__data_start__` 系）

CMSIS の C スタートアップ実装（`startup_ARMCM0plus.c` 等）で使われる
`__data_start__` / `__data_end__` / `__bss_start__` / `__bss_end__` /
`__StackTop` / `__StackBottom` / `__etext` / `__VECTOR_TABLE` を使う。

利点:
- 実務標準であり、CMSIS や各ベンダのスタートアップコードと整合する
- 既存の crt0 実装例を参照・比較しやすい（学習価値・移植性）
- ハンドラ命名（`Reset_Handler` 等）が既にCMSIS準拠であり、体系が統一される

欠点:
- 学習ノートの教科書的名前と異なるため、対応関係の説明が別途必要

### 判断2: データ初期化方式

#### 選択肢2-A: 単純シンボル方式（`__data_start__` / `__data_end__` / `__etext`）

`.data` の実体範囲（VMA）を `__data_start__`〜`__data_end__`、初期値の
格納位置（LMA）を `__etext` として提供し、crt0 は1つの連続領域を
コピーする前提で実装する。

利点:
- crt0 の実装が単純（1領域のコピーループと1領域のゼロクリア）
- 本プロジェクトのメモリ構成（単一RAM領域）に過不足なく合致する
- 理解しやすく、学習段階の crt0 に適する

欠点:
- 複数の初期化領域（例: 別々のRAMバンクに配置する `.data` の分割、
  特定セクションのRAM常駐など）が必要になった場合、対応できない

#### 選択肢2-B: テーブル駆動方式（CMSIS `__copy_table_*` / `__zero_table_*`）

CMSIS が採用する、コピー元・コピー先・サイズの三つ組をテーブル化し、
crt0 がテーブルを走査して任意個数の領域を初期化する方式。

利点:
- 複数の初期化領域に対応できる（拡張性が高い）
- CMSIS 標準の crt0 をほぼそのまま利用できる

欠点:
- 単一RAM領域しかない本プロジェクトには過剰
- crt0 とリンカスクリプトの両方が複雑になり、学習段階での見通しが悪い

### 判断3: メモリ領域の宣言サイズ

#### 選択肢3-A: 実サイズで宣言

FLASH = 2048k、RAM = 264k と、RP2040 の実際のメモリサイズで宣言する。

利点:
- シンプルで、データシートのメモリマップと1対1で対応する
- 利用可能な全メモリを使える

欠点:
- ブートローダ用の領域予約、SRAMの用途別分割（コア別スタック、
  DMA専用バッファ領域など）が必要になった場合は見直しが要る

#### 選択肢3-B: 用途を見越した分割宣言

将来の領域予約やSRAM分割を最初から MEMORY ブロックに織り込む。

利点:
- 後からの構成変更が局所化される可能性

欠点:
- 現時点で必要のない分割を先取りすることになり、根拠のない境界が固定化
  される（YAGNIに反する）
- RP2040のSRAMは6バンク構成だが、本プロジェクトでバンク分割を要する
  要件は現時点で存在しない

## 決定

- 判断1: **選択肢1-B（CMSIS準拠命名）** を採択する。
- 判断2: **選択肢2-A（単純シンボル方式）** を採択する。
- 判断3: **選択肢3-A（実サイズ宣言）** を採択する。

学習ノートの教科書的シンボル名（`_sdata` 系）は、説明の分かりやすさを
優先して **学習ノート側ではそのまま維持** し、実装シンボルとの対応関係を
`rp2040.ld` 内のコメントに記載する（命名体系を実装とノートで意図的に
分離する）。

## 理由

3つの判断はいずれも「実務標準への整合」と「学習段階に見合うシンプルさ」
という本プロジェクトの基本方針に沿って決定した。

1. **命名（1-B）**: ハンドラ名が既にCMSIS準拠であり、シンボルも揃える
   ことで体系が一貫する。実務標準に触れること自体がプロジェクトの学習
   目的（単一企業の慣習に偏らない業界標準の習得）に合致する。学習ノートの
   教科書的名前は概念理解には優れるため、ノート側は変更せず対応表で橋渡し
   する。

2. **初期化方式（2-A）**: 本プロジェクトは単一RAM領域であり、テーブル駆動の
   拡張性を必要としない。crt0 を単純に保つことが段階2-C-4の学習価値を高める
   （YAGNI原則）。

3. **メモリ宣言（3-A）**: 現時点で領域予約・SRAM分割を要する要件がない。
   データシートのメモリマップと素直に対応させ、必要が生じてから見直す。

## 影響

ポジティブな影響:
- crt0（段階2-C-4）の実装が単純な1領域コピー + 1領域ゼロクリアで済む
- CMSISや各ベンダのスタートアップ実装を参照・比較しやすく、学習・将来の
  移植に有利
- シンボル体系がハンドラ命名と統一され、コード全体の一貫性が保たれる

ネガティブな影響（技術的負債）:
- 複数の初期化領域やSRAMバンク分割が必要になった場合、リンカスクリプトと
  crt0 の両方を見直す必要がある（単純シンボル方式 → テーブル駆動方式への
  移行、MEMORYブロックの分割）
- 学習ノートと実装でシンボル名が異なるため、読者は対応表を参照する必要が
  ある

将来の見直し条件:
- 特定セクションのRAM常駐や、複数RAM領域への `.data` 分割が必要になった時
  → 判断2をテーブル駆動方式（2-B）へ移行
- コア別スタックやDMA専用領域などSRAM分割が必要になった時
  → 判断3の MEMORY ブロックを分割宣言（3-B）へ見直し
- Rust移植時（第2フェーズ）に `cortex-m-rt` 等のクレートが規定する
  リンカ規約・シンボル名を採用する場合 → 判断1・2を再検討

## 補足: ベクタテーブルの256バイト境界配置（段階2-C-3-4で追加）

段階2-C-3-4のリンカスクリプト検証において、`.vectors` セクションの配置が
boot2 の実バイト数に依存して変動する問題が判明したため、本決定を補足する。

### 文脈

検証時点の boot2 は CRC32 チェックサム未付加で 240 バイトであり、当初の
リンカスクリプトでは `.vectors` が boot2 の直後（0x100000F0）に詰めて
配置されていた。しかし以下の2点から、この配置は不正である。

- RP2040 の boot2 は「実コード 252 バイト + CRC32 4 バイト = 256 バイト」の
  固定領域である。CRC 付加後は 256 バイトとなり、`.vectors` の位置が動く。
- Cortex-M0+ の VTOR（Vector Table Offset Register）は下位 8 ビットが無効で
  あり、ベクタテーブルは 256 バイト境界に配置しなければならない。
  0x100000F0 は 256 の倍数でないため、VTOR に正しく設定できない。

### 決定

`.boot2` セクションを boot2 の実サイズに関わらず 256 バイト固定で確保し
（`. = __boot2_start__ + 256;`）、後続の `.vectors` が常に 0x10000100
（256 バイト境界）から始まるようにする。

### 理由

- boot2 が 256 バイト固定領域であることは RP2040 の根本仕様であり、
  リンカスクリプト上で明示することで意図が明確になる
- ベクタテーブルの位置が boot2 の実装状態（CRC 付加の有無）に依存しなくなる
- VTOR の 256 バイト境界制約を満たす
- pico-sdk の標準リンカスクリプトも boot2 を専用領域として確保しており、
  実務標準と整合する（ADR-0003 本体の CMSIS 準拠方針と一貫）

### 影響

- crt0（段階2-C-4）は VTOR に 0x10000100 を設定でき、ベクタテーブルを
  正しく参照できる
- boot2 の CRC32 付加（256 バイト化）の前後で `.vectors` 以降の配置が
  変わらない

### 検証

段階2-C-3-4 にて、`make verify-linker` で `.vectors` が 0x10000100 に
配置されることを readelf / objdump で確認済み。

## 補足: boot2 チェックサムの付加（段階2-C-3-5）

### 背景

RP2040 のブート ROM は、フラッシュ先頭 256 バイト(boot2)の先頭 252 バイトに対する
CRC-32/MPEG-2 を、末尾 4 バイト(オフセット 252–255、リトルエンディアン)と照合する。
一致しなければ boot2 を実行せず、USB ブートローダ(RPI-RP2 ドライブ)としてマウント
される。本リンカスクリプトは `.boot2` を 256 バイトに固定済みだが、末尾 4 バイトは
0x00 のままであり、この状態で焼くと bootrom の照合に必ず落ちる。本段階でその 4 バイトを
正しい CRC で満たし、boot2 を「焼ける・bootrom が受け入れる」成果物にする。

### 決定

1. CRC アルゴリズム: CRC-32/MPEG-2 を採用する(poly=0x04C11DB7, init=0xFFFFFFFF,
   入出力ビット反転なし, 最終 XOR なし)。RP2040 Datasheet 2.8.1.3.1 に従う。実装の
   正しさは規格の標準チェック値 `crc32_mpeg2("123456789") == 0x0376E6E7` で自己検証する。

2. チェックサムの配置: 256 バイトの末尾 4 バイトに、リトルエンディアンで格納する。

3. 生成方法: 最終ファーム全体を後処理する方式ではなく、boot2 を単独で処理する。
   boot2 は内部関数 read_flash_sreg への bl(R_ARM_THM_CALL 再配置)を含むため、
   boot2.o を単独リンク(`ld --section-start .boot2=0x10000000`)して再配置を解決した
   boot2.elf を作り、そこから `objcopy -O binary -j .boot2` で raw バイト列を抽出する。
   これを `pad_checksum.py` が 252 バイトまで 0x00 パディングして CRC を付加し、
   256 バイトの `boot2_checksummed.bin` を出力する。(pico-sdk のようにチェックサム済みの
   `.S` を生成して再リンクする方式は採らない。)

4. 検証手段: `pad_checksum.py --check` で、サイズ(256)と末尾 CRC の一致を確認し、
   不一致時は非ゼロ終了する。これにより将来 CI(INFRA-5)でビルド成果物として機械的に
   検証できる。

### 帰結

- boot2 が bootrom に受理される 256 バイトの正当なブロブになり、`.vectors` を
  「boot2 は 256 バイト」前提で 0x10000100 に固定した決定が、実体としての 256 バイト
  イメージに裏付けられる。
- boot2 は read_flash_sreg への内部 bl(R_ARM_THM_CALL)を含むため、未リンクの .o から
  直接 objcopy すると bl が未解決(プレースホルダ)のまま抽出されてしまう。単独リンクで
  解決してから抽出する。boot2 には絶対アドレス再配置(R_ARM_ABS32)が無く bl は PC 相対
  のため、リンクの基底アドレス(ここでは 0x10000000)に依らず .boot2 のバイト列は同一に
  なる。リンク後は `readelf -r build/boot2.elf` が「リロケーション無し」を示す。
- 本段階の検証は静的検証(サイズ・CRC 一致)に留まる。実機での起動受理の確認は
  段階 2-D-2(LED 点滅)で行う。
- boot2 を最終ファームウェアイメージへ統合する方式(チェックサム済みブロブを別
  セクションとしてリンクするか、最終 .bin を後処理パッチするか)は、crt0 と全体
  イメージが揃う段階 2-C-4 / 初回フラッシュ時に決定する。
