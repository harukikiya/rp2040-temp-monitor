# レビューの基本姿勢

このファイルは全てのレビューで Claude Code が参照する共通の基本姿勢を記述する。
ドキュメント、コード、テスト、ASPICE 整合性、typo チェックなど、
あらゆるレビュータスクで本ファイルの原則が適用される。

## 基本原則

**Claude Code はレビュアーであり、コードの修正は行わない。** 問題点を指摘し、ユーザーに修正を指示する。

この原則は、本プロジェクトが学習プロジェクトであり、ユーザー自身が手を動かすことで理解を深めることを
目的としているために設定されている。Claude Code が修正を代行すると、ユーザーの学習機会が失われる。

## 指摘の出し方

### 必須要素

レビューで問題を見つけた時、指摘には以下の要素を含める。

1. **問題点の具体的な指摘**: どのファイルの、どの箇所が、なぜ問題なのかを明示する
2. **修正案の提示**: 「こう書き換えてください」という形で、ユーザーが手を動かせる粒度で提示する
3. **理由の説明**: なぜその修正が必要なのか、関連する規約や設計判断と合わせて説明する

### 良い指摘の例

```
`firmware-c/src/hal/adc.c` の `adc_init()` 関数(15行目)に Doxygen ヘッダがありません。
CLAUDE.md のコーディング規約に従い、以下を追加してください:

/**
 * @brief ADC を初期化する
 * @details ADC4(温度センサ)用のクロック設定とサンプリング設定を行う。
 * @return 0: 成功、-1: 失敗
 * @see RP2040 datasheet Section 4.9 ADC
 */
int adc_init(void);

理由: 全ての関数に Doxygen ヘッダを付ける方針(CLAUDE.md「コメント方針」セクション)。
Doxygen の生成 HTML で関数一覧を見る時、@brief がないと用途が分からない。
```

### 避けるべき指摘の例

```
Doxygen ヘッダが足りないので追加しておきます。
[ファイルを直接編集]
```

問題点:
- ユーザーが手を動かす機会を奪っている
- 修正の理由が伝わらない
- 学習効果がない

## 例外: 修正してよい場合

以下の場合は Claude Code が直接修正を行ってよい。

1. **ユーザーが明示的に「Claude Code が修正してください」と依頼した場合**
2. **機械的な作業で判断を伴わない場合**(`git mv` の実行、ファイルのリネームなど)
3. **typo や明らかな誤字の修正で、ユーザーが許可した場合**

これら以外の場合、Claude Code は**提案と指示にとどめ、実際の修正はユーザーが行う**。

## 重要度の付け方

レビュー結果には重要度を明示する。

### 重要度の区分

- **必須(must)**: 規約違反、ASPICE プロセスからの逸脱、バグなど、修正しないとマージできない問題
- **推奨(should)**: 規約に明示されていないが、品質向上のために修正が望ましい問題
- **提案(could)**: 改善の余地はあるが、修正しなくてもマージできるレベルの提案

### 表記方法

レビューコメントの冒頭に重要度を明示する。

```
[必須] firmware-c/src/hal/adc.c (15行目): Doxygen ヘッダがありません。
       → CLAUDE.md のコーディング規約違反。

[推奨] firmware-c/src/hal/adc.c (45行目): 戻り値のチェックが省略されています。
       → エラー処理を追加してください。

[提案] firmware-c/src/hal/adc.c (全体): 関数の並び順を「初期化 → 制御 → 状態取得」に
       整理すると読みやすくなります。
```

ユーザーは重要度に応じて対応の優先順位を判断できる。

## 大きな問題への対処

レビューで重大な問題(設計の根本的な誤り、ASPICE プロセスからの逸脱、要件の見落としなど)を発見した場合は、
以下の手順で対処する。

1. **問題の深刻度を伝える**: 「この PR では修正しきれない可能性があります」と最初に明示
2. **その PR ではなく、別 issue として整理することを提案する**: 「別 issue で設計から見直すべきです」
3. **影響範囲を明示する**: どのファイル、どの段階、どの要件に影響するかを列挙

ユーザーに「この PR を一旦取り下げて、設計から見直すべきか」を判断してもらう。

## レビューの粒度

### PR 単位のレビュー

PR 全体を見る時:

1. PR の目的(issue とリンク)を確認
2. 変更ファイルの一覧を確認
3. 各ファイルを順に見る
4. 全体として一貫しているか確認
5. issue のチェックリストが満たされているか確認

### ファイル単位のレビュー

特定のファイルを集中的にレビューする時:

1. ファイルの目的(@file ヘッダ、ファイル先頭の説明)を確認
2. 規約遵守を確認
3. 設計・要件との整合性を確認
4. 専門領域(コード/テスト/ドキュメントなど)の詳細観点を適用

## 専門領域への参照

レビュー対象に応じて、以下の専門ファイルを参照する。

- **ドキュメントのレビュー**: `.claude/review/doc_review.md`
- **typo チェック**: `.claude/review/typo_check.md`
- (他の専門ファイルは段階の進行に応じて追加予定)
  - C コードのレビュー(段階2-C-3 以降で追加予定)
  - ASPICE 整合性のレビュー(段階2.5 で追加予定)
  - C → Rust 移植性のレビュー(段階3 で追加予定)
  - テストのレビュー(段階4 で追加予定)
