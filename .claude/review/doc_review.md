# ドキュメントレビュー

このファイルは Markdown 形式のドキュメント(`docs/` 配下、`README.md`、`CONTRIBUTING.md` など)を
レビューする際の観点と手順を記述する。

レビューの基本姿勢は `.claude/review/principles.md` を参照。

## 観点

### 1. Markdown 構文

- リンクが切れていないか(他ファイルへの相対パス、外部URL)
- 見出し階層が論理的か(h1 → h2 → h3 の順、レベルの飛び越しなし)
- コードブロックの言語指定があるか(```c、```python など)
- リストのインデントが揃っているか
- 表が壊れていないか(列数の不一致など)

### 2. MyST 拡張(Sphinx)

- `{toctree}` の参照先ファイルが存在するか
- `{uml}` ディレクティブのパスが正しいか(`.puml` ファイルの存在)
- `{note}`、`{warning}` などのディレクティブが正しく閉じられているか
- 内部参照(`:ref:`、`:doc:`)が有効か

### 3. sphinx-needs

- 要件(SWR、SYS など)の ID 形式が統一されているか(`SWR_NNN` 形式)
- `:links:` のリンク先が存在するか(SWR から SYS、ADR から SWR など)
- `:status:` が定義された値か(`draft`、`approved` など)
- `:type_kind:` が定義された値か(`Functional`、`Performance`、`Safety` など)
- 同一 ID の重複がないか

### 4. ASPICE プロセス対応

各ドキュメントファイルの冒頭に「位置づけ」セクションが必要(段階2.5 以降)。

- 「位置づけ」セクションが存在するか
- ASPICE のどのプロセス(SYS.2、SWE.1 など)に該当するか明示されているか
- 関連する他文書への参照があるか

### 5. ファイル番号体系

- ファイル名が ASPICE プロセス順の 10刻みの番号体系に従っているか
  - `00_overview.md`
  - `10_system_requirements.md` (SYS.2)
  - `20_software_requirements.md` (SWE.1)
  - `30_architecture.md` (SWE.2.1)
  - `40_component_design.md` (SWE.2.2)
  - `50_detailed_design.md` (SWE.3.1)
  - `60_integration_test_design.md`
  - `70_component_test_design.md`
  - `80_unit_test_design.md`
  - `99_traceability.md`

### 6. PlantUML 図の参照

- 図が必要な箇所で使われているか(アーキテクチャ、ブートシーケンス、フローなど)
- 図のパス(`diagrams/<カテゴリ>/<名前>.puml`)が正しいか
- 図のタイトル、説明が本文と整合しているか

### 7. ADR への参照

設計判断が文書化されている時、ADR への参照があるか。
逆に、本文に大きな設計判断が書かれているのに ADR がない場合、ADR 作成を提案する。

### 8. 文体

- 日本語の文章は「である調」で統一されているか
- 専門用語が必要に応じて補足されているか
- 読み手(学習プロジェクトの場合、新人レベルを想定)に対して分かりやすいか

### 9. クロスリファレンス

- 他文書への言及がある時、リンクが付いているか
- 「上記」「下記」などの相対表現が、文書構造の変更で壊れていないか

### 10. 整合性

- 同じ事実が複数箇所に書かれている時、内容が一致しているか
- 進捗ステータス(README.md)と各文書の状態が一致しているか
- 段階構成が CLAUDE.md と一致しているか

## 手順

### PR 全体のレビュー

1. PR 内の `.md` ファイルを全て確認
2. 各ファイルに対して、上記「観点」を適用
3. ファイル間の整合性をチェック(`toctree`、相互参照)
4. Sphinx ビルドが通る前提なので、ビルドエラーは出ない想定だが、警告レベルの問題は指摘

### 単一ファイルのレビュー

1. ファイル冒頭の「位置づけ」セクションを確認(段階2.5 以降)
2. 見出し階層を確認
3. 本文を順に読み、観点1〜10 を適用
4. リンク、参照を一つずつ追跡

## 出力フォーマット

レビュー結果は `principles.md` の重要度区分(必須/推奨/提案)に従って整理する。

```
## 必須

- [ファイル名:行番号] 問題点
  → 修正案
  → 理由

## 推奨

- [ファイル名:行番号] 問題点
  → 修正案
  → 理由

## 提案

- [ファイル名:行番号] 改善点
  → 提案
```

## このファイルの更新方針

ドキュメントレビューの観点や手順を更新する時は、このファイルを編集する。
プロジェクトの段階が進むにつれて、新しい観点(例: 段階2.5 で「テスト設計のレビュー観点」を追加)が
必要になる場合がある。その時は本ファイルに追記する。

更新は `[INFRA-N]` issue でPR として行う。
